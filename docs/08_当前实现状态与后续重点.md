# 当前实现状态与后续重点

## 当前已经基本具备的内容

- 本地成就系统、成就提示弹窗，以及后续 Steam 成就适配边界
- 窗口/全屏显示设置，窗口模式下固定 16:9 比例
- 地图边界显示、玩家边界限制、右下角小地图
- 统一幸存者风格 UI 组件：弹窗、卡牌列表、悬停详情、卡片工厂
- 主菜单设置面板居中/自适应修复
- 升级界面卡牌短摘要 + 悬停详情，大招/普攻/独立被动详情说明
- 主题式 Build 卡池：默认三相荡阵，按配比解锁万向锋路 / 血盾回路 / 三相终式
- 英雄特性训练与共同致富：三英雄特性成长、切人冷却乘算缩减

- 主菜单
- 存档位选择
- 主线准备界面功能版
- 战斗主循环
- 三角色切换
- 普攻 / 进场技 / 退场技 / 大招
- 经验与升级
- Build 菜单
- 普通怪 / 精英 / Boss 基础框架
- HUD / 暂停菜单 / 游戏结束界面
- BGM 与继续游戏
- 开发者模式

## 当前已验证的主爽点

- 切人节奏成立
- 大招演出能形成明显高潮
- 三个角色已有基本差异
- 项目已经脱离“只有概念没有手感”的阶段

## 当前主要短板

### 构筑深度仍然需要继续扩展

- 主题式 Build 骨架已经建立，但目前主要围绕“三相荡阵”一组核心主题和三个分支展开
- 后续需要继续补更多主题、更多跨主题配比，以及更明确的视觉差异
- 新增卡必须同步补齐三英雄名称、三英雄数值说明和实际战斗效果

### 构筑与角色特征仍需继续绑定

- 三个角色必须继续强化各自的独特演出语言
- 剑士应该更像夸张斩击体系
- 枪手应该更像夸张弹幕体系
- 术师应该更像轰炸和区域控制体系

### 工程复杂度已经明显上升

- `player.gd` 和 `enemy.gd` 仍然是高风险枢纽
- 本轮已进一步抽出 `scripts/player/player_theme_skill_flow.gd` 和 UI 通用组件，但 Build 触发、普攻、大招、存档仍需要谨慎验证
- 继续堆功能会降低后续维护效率

## 当前适合优先推进的方向

1. 在现有主题式 Build 骨架上继续加第二组主题或跨主题进化
2. 继续加强构筑带来的视觉差异，特别是新被动触发时的演出辨识度
3. 完善主线准备界面和故事模式骨架
4. 继续丰富怪物生态和 Boss 剧本
5. 设计第 4、第 5 个角色前，先决定 Build 详情是否从固定三英雄表升级为“当前队伍角色表”

## 当前不建议立刻做的事情

- 大规模重构全部代码
- 一口气做完整剧情演出系统
- 一开始就做最终美术成品规格
- 把所有未验证系统同时并行推进

## 推荐的下一阶段开发顺序

1. 稳住当前战斗可玩版本
2. 继续打磨 Build 深度与角色差异
3. 把主线准备和关卡推进流程跑通
4. 继续扩展关卡内容和难度模式
5. 再进入更系统的剧情、营地互动和长期养成

## 结论

- 项目已经不是“做不出来”的阶段
- 当前核心任务不是从零开始想，而是把现有系统持续整理、扩展、去同质化
- 这套文档的目的，就是让后续继续开发时不需要反复从聊天记录里捞结论

## 本轮交接重点（2026-05-01）

- `scripts/build/build_database.gd` 是当前 Build 主题、卡牌类型、角色差异文案和数值说明的权威数据源。
- `scripts/build/build_system.gd` 负责主题解锁、卡池生成、卡片 payload 拼装和三英雄详情文本。
- `scripts/player/player_attribute_flow.gd` 与 `scripts/player/roles/role_attribute_rules.gd` 负责英雄特性训练、旧存档迁移、共同致富计数和切人冷却倍率。
- `scripts/player/player_theme_skill_flow.gd` 负责主题卡触发型被动和独立冷却被动。
- `scripts/player/player_skill_cooldown_slots.gd` 与 `scripts/player/player_ultimate_flow.gd` 负责把普攻/大招强化折叠进悬停说明。
- `scripts/ui/*` 下的通用组件已经成为升级界面、暂停/结算、存档/难度、主菜单设置的共享 UI 基础。
- 已新增 `scripts/tests/ui_build_handoff_smoke.gd`，覆盖主题解锁、角色差异卡名、设置弹窗居中、升级悬停去重、技能栏标题和大招名称。
