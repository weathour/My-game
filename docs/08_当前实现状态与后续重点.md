# 当前实现状态与后续重点

## 当前阶段

项目当前使用祝福驱动的升级和技能解锁体系。

## 已具备内容

- 主菜单、存档选择、无尽模式、暂停菜单、HUD、角色面板和开发者模式。
- 三角色战斗：剑士、枪手、术师。
- 自动攻击 / 鼠标跟随攻击模式。
- 普攻、后台攻击、切人、大招、技能冷却栏。
- 祝福三选一升级、祝福等级、祝福合成、祝福技能解锁。
- 小 Boss 装备奖励、装备持有和角色间转赠。
- 地图边界内波次刷怪、刷怪警告、难度 profile、性能保护、高密度战斗性能证据门禁和新怪入场削峰队列。
- 本地成就和自动检查脚本。

## 当前主要风险

- `player.gd` 仍是高风险协调脚本，新增功能应继续下沉到 `scripts/player/*`。
- `main.gd` 仍承担战斗主控，新增流程应优先放入 `scripts/game/*`。
- 祝福、技能、装备、角色面板和存档之间的数据边界需要持续验证。
- 大量特效命中大量敌人、或同一批刷怪警告同时结束并实例化新怪时，仍可能产生性能压力；后续优化需要继续用同场景 p95 / p99 / max 帧时间、CPU/core 证据和玩法等价计数作门禁。

## 后续重点

1. 继续补全祝福驱动技能的实际战斗表现。
2. 调整祝福数值和高等级刷新概率。
3. 完善角色面板里祝福、技能、道具和数值的可读性。
4. 继续优化大量命中判定和特效触发时的性能，并保持 benchmark / evaluator / feature flag 证据链先行。
5. 扩展小 Boss / Boss 奖励，但保持奖励逻辑模块化。
6. 主线模式暂不开放，但保留相关代码。

## 当前开发护栏

- 新增进度内容必须进入祝福、技能或装备模块。
- 不新增独立进度目录来绕过当前祝福体系。
- 运行代码入口命名必须反映当前祝福、技能、装备和奖励职责。
- 新功能如果属于祝福，优先改 `player_blessing_system.gd` 和 `player_blessing_skill_state.gd`。
- 新功能如果属于奖励流程，优先改 `game/reward_flow.gd`、`player_reward_applier.gd` 或 `player_equipment_flow.gd`。
