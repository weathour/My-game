# 技术债、风险点与重构建议

## 一、当前最大的技术债

### 1. `player.gd` 过大

当前 `player.gd` 已经承担了太多职责：

- 玩家移动
- 三角色轮转
- 普攻
- 大招
- Build
- 属性成长
- 角色风格
- 特效生成
- 命中判定
- 各种临时状态

风险：

- 修改一个功能时容易误伤别处
- 很难快速定位某个问题到底属于哪一类系统
- 新功能继续追加会越来越难维护

### 2. `enemy.gd` 也在持续膨胀

当前 `enemy.gd` 同时负责：

- 普通怪
- 精英怪
- Boss
- Boss 各阶段与多种技能

风险：

- 普通怪调整容易影响 Boss
- Boss 继续扩展时会越来越难读

### 3. 数据与逻辑耦合较重

目前很多内容直接写在脚本里：

- 角色卡池
- 关卡参数
- 敌人参数
- 样式参数
- 文案

风险：

- 策划调整成本高
- 很多参数只能改代码
- 后面很难做大规模平衡表

## 二、当前最容易出问题的区域

### 1. Build 系统

原因：

- 既涉及 `player.gd`
- 又涉及 `build_database.gd` / `build_system.gd`
- 又涉及 `level_up_ui.gd`、HUD、hover detail
- 又涉及文案、数值、特效和存档

典型风险：

- 文案和效果不一致
- 三英雄对应效果写了两份或显示重复
- 当前站场角色显示名和实际效果不一致
- 主题解锁配比与卡池生成不同步
- 最终效果卡进度和真实判定不同步
- 无独立冷却的被动误塞进技能栏，导致 HUD 混乱

### 2. 特效与判定

原因：

- 特效在资源里改
- 判定在代码里改
- 容易一边变了，另一边没跟上

典型风险：

- 看起来打中了但没有伤害
- 方向逻辑正确但旋转中心错误
- 换特效后又重踩旧坑

### 3. 继续游戏 / 存档恢复

原因：

- 涉及 `main.gd`
- `save_manager.gd`
- BGM 状态
- 玩家 Build、计时器、角色外观

典型风险：

- 倒计时重置
- Build 等级错乱
- 角色贴图 / 颜色恢复失败
- BGM 进度异常

## 三、当前建议的轻量重构顺序

注意：不是现在立刻大重构，而是之后边做边拆。

### 第零优先级：守住已经建立的边界

本轮已经先建立了一个小边界：

- `main.gd` 不再直接调用 `AchievementService`
- 战斗成就事件统一经由 `scripts/game/game_achievement_bridge.gd`
- 敌人击败后的奖励/成就/HUD 分支统一经由 `scripts/game/enemy_defeat_flow.gd`
- 战斗场景 UI 创建和玩家信号连接统一经由 `scripts/game/game_scene_wiring.gd`
- HUD 投影、Boss UI、性能指标显示统一经由 `scripts/game/game_hud_flow.gd`
- 角色面板打开/关闭和暂停规则统一经由 `scripts/game/game_character_panel_flow.gd`
- 地图边界显示、小地图数据采集统一经由 `scripts/game/game_map_flow.gd`，玩家边界限制经由 `scripts/player/player_map_bounds_flow.gd`
- 故事/无尽上下文读取、故事配队应用和关卡参数派生统一经由 `scripts/game/game_story_context_flow.gd`
- Steam/GodotSteam 只允许出现在 `scripts/achievements/steam_achievement_adapter.gd`
- `.graphifyignore` 排除 `addons/godot_mcp/`，避免架构图被编辑器工具插件污染
- UI 通用基础已抽到 `scripts/ui/theme/`、`scripts/ui/core/`、`scripts/ui/components/`
- 主题卡触发逻辑已抽到 `scripts/player/player_theme_skill_flow.gd`
- 英雄特性训练已集中到 `scripts/player/player_attribute_flow.gd` 与 `scripts/player/roles/role_attribute_rules.gd`
- `scripts/tests/check_architecture_contract.py` 会自动检查核心架构约定
- `scripts/tests/ui_build_handoff_smoke.gd` 会自动检查主题解锁、设置弹窗居中、悬停详情去重和 HUD 技能栏显示约定

后续接新成就、平台 SDK、统计系统时，应优先延续这种“小桥接层 + 自动检查”的方式。

### 第一优先级

把 `player.gd` 中的逻辑逐步按职责拆成辅助脚本或清晰分区：

- 角色战斗核心
- Build 数据与应用
- 特效生成
- 队伍轮转与进退场

### 第二优先级

把 `enemy.gd` 的 Boss 行为逐步抽离：

- 普通敌人行为
- 精英额外行为
- Boss 阶段与技能控制

### 第三优先级

把静态数据逐步外移：

- 角色卡数据
- 关卡数据
- 样式数据
- 敌人参数表

## 四、当前不建议做的重构

- 全项目一次性推倒重写
- 为了“架构优雅”而暂停所有内容开发
- 在还没确定玩法方向前把所有东西彻底抽象成复杂框架

## 五、当前正确做法

- 继续先做可玩的内容
- 但每次碰到高频修改区域时，顺手做小拆分
- 目标是：
  不阻碍开发速度
  又逐步减少后续爆炸风险

## 六、给后续继续开发的提醒

- 角色特效和判定一定同步验证
- Build 文案与实际效果一定同步验证
- 继续游戏必须作为高风险功能看待
- 任何“只是临时加一下”的逻辑，如果写进 `player.gd`，都要意识到它后面几乎一定会继续膨胀
- 任何平台能力，例如 Steam 成就、Steam 云存档、排行榜，都不要直接写入 `main.gd` / `player.gd`
- 每次建立新的架构边界，都应补一个最小自动检查，防止后续无意打破

## 七、当前 Build/UI 新增护栏

- 新增主题时先改 `build_database.gd`，再补 `build_system.gd` 和触发 flow，不要只改 UI 文案。
- 新增卡必须给三名英雄都补 `ROLE_CARD_VARIANTS` 和 `CARD_ROLE_NUMBERS`。
- 新增无独立冷却的强化时，优先补到普攻/大招 hover 说明，避免占用技能栏。
- 修改设置弹窗或新建弹窗时，不要回到固定 `position` 或 `CenterContainer` 方案；优先使用 `SurvivorsModal` 全屏遮罩 + 居中 panel。
- 修改 hover detail 时必须验证“三英雄对应效果 / 数值”不会出现两次。
- 修改共同致富或英雄特性时，必须验证旧存档迁移键：`strength/agility/intelligence`、`vitality/power`、`swordsman_trait/gunner_trait/mage_trait`。
