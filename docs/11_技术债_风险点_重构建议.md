# 技术债、风险点、重构建议

## 高风险区域

- `player.gd`：仍是玩家状态和战斗桥接中心，新增功能不能继续堆在这里。
- `main.gd`：战斗主控仍较重，新增流程应进入 `scripts/game/*`。
- 祝福系统：祝福数据、技能解锁、角色面板、存档和 HUD 都依赖同一份状态。
- 特效和命中判定：大量敌人同时被多段技能命中时可能造成掉帧。
- 存档兼容：旧字段迁移和当前祝福/装备状态需要持续验证。

## 当前护栏

- 后续新增进度内容必须进入祝福、技能或装备模块。
- 祝福数据只放在 `player_blessing_system.gd`。
- 技能解锁和技能倍率只放在 `player_blessing_skill_state.gd`。
- 装备奖励只放在 `player_equipment_flow.gd`。
- 小 Boss / Boss 奖励流程放在 `reward_flow.gd` 和对应 player applier。
- HUD 显示通过 `game_hud_flow.gd` 和 HUD 子组件，不从玩法代码直接拼 UI。

## 新功能落点

- 新祝福：`scripts/player/player_blessing_system.gd`。
- 新祝福技能：`scripts/player/player_blessing_skill_state.gd` + `scripts/abilities/*_ability.gd`。
- 新角色普攻：`scripts/player/roles/*_role.gd`。
- 新装备：`scripts/player/player_equipment_flow.gd`。
- 新奖励流程：`scripts/game/reward_flow.gd`。
- 新刷怪规则：`scripts/game/enemy_spawn_flow.gd` 或 `scripts/enemy/enemy_director.gd`。
- 新角色面板信息：`scripts/ui/hud/character_panel.gd`。

## 验证建议

- 修改祝福后跑 `player_blessing_system_smoke.gd`。
- 修改架构边界后跑 `check_architecture_contract.py`。
- 修改 GDScript 后跑 Godot headless parse。
- 修改角色面板或奖励 UI 后至少手动检查一次升级菜单、小 Boss 奖励和 C 面板。

## 不建议

- 不建议把废弃设计文档里的机制重新搬回运行代码。
- 不建议为了短期实现把技能逻辑直接写进 `player.gd`。
- 不建议让 UI 文案和实际数值分离。
- 不建议绕过 `player_upgrade_applier.gd` 直接在 UI 里改玩家状态。
