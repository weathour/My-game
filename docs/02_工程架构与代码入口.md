- 配置文件：`project.godot`

## 目录结构

- `assets/`
  存放菜单背景、角色贴图等资源
- `effects/`
  存放特效场景与特效贴图，例如剑士斩击 `slasheffect3`
- `scenes/`
  存放主要游戏场景、UI 场景、战斗单位场景
- `scripts/`
  存放核心逻辑脚本
- `data/`
  存放可数据化的项目定义，例如本地成就定义
- `shaders/`
  存放材质与 shader
- `ui/`
  预留 UI 资源目录

## 主要脚本职责

- `scripts/main.gd`
  战斗主控。负责关卡时间、刷怪、精英/Boss 触发、UI 装配、继续游戏、自动存档。

- `scripts/player.gd`
  绝对核心脚本。负责玩家移动、三角色状态、普攻、切人、进场技、退场技、大招、Build、HUD 数据、特效触发、角色样式等。

- `scripts/enemy.gd`
  敌人基础行为、成长、精英/Boss 模式切换、Boss 技能逻辑。

- `scripts/enemy_bullet.gd`
  敌方弹幕行为。

- `scripts/bullet.gd`
  我方子弹行为，主要服务枪手相关攻击。

- `scripts/hud.gd`
  负责时间、等级、HP、MP、经验条、Boss 血条、队伍信息等显示。

- `scripts/level_up_ui.gd`
  负责升级菜单、英雄特性训练 + Build 卡双选择、卡牌悬停详情、小 Boss 奖励双选。

- `scripts/build/build_database.gd`
  负责主题式 Build 数据、卡牌类型、三英雄卡名、三英雄效果/数值说明。

- `scripts/build/build_system.gd`
  负责主题解锁、卡池生成、卡牌 payload、奖励选项和旧 ID 兼容。

- `scripts/player/player_attribute_flow.gd` / `scripts/player/roles/role_attribute_rules.gd`
  负责英雄特性训练、共同致富、旧存档迁移和相关数值规则。

- `scripts/player/player_theme_skill_flow.gd`
  负责主题卡触发效果和独立冷却被动。

- `scripts/ui/core/survivors_modal.gd`、`scripts/ui/components/`、`scripts/ui/theme/survivors_ui_theme.gd`
  负责通用弹窗、卡牌列表、悬停详情、卡片工厂和幸存者风格主题。

- `scripts/pause_menu.gd`
  ESC 菜单逻辑，包括继续、重开、返回主菜单、音量设置。

- `scripts/main_menu.gd`
  主菜单逻辑，负责开始、继续、设置、开发者模式入口、菜单 BGM。

- `scripts/save_manager.gd`
  存档系统。负责故事存档、战斗存档、继续游戏目标、元数据。

- `scripts/story_data.gd`
  主线数据定义。包括关卡表、角色池、风格样式、默认故事档结构。

- `scripts/story_prep.gd`
  主线准备界面。负责当前编队、角色顺序、风格切换、进入下一关。

- `scripts/developer_mode.gd`
  开发者模式开关，控制无敌、无限大招、全卡候选、禁用存档等。

- `scripts/bgm_player.gd`
  BGM 播放器逻辑，负责音量配置、静音、暂停、续播位置。
- `scripts/game/game_map_flow.gd`
  战斗地图展示桥接。负责创建地图边界显示，并把玩家、敌人、Boss、掉落物位置投影给 HUD 小地图。
- `scripts/map/map_boundary_view.gd`
  世界空间地图边界绘制节点。
- `scripts/player/player_map_bounds_flow.gd`
  玩家移动后的地图边界限制，防止玩家离开当前战斗地图范围。

## Autoload 服务

当前 `project.godot` 注册了这些全局服务：

- `AchievementService`
  本地成就权威状态源，读取 `data/achievements.json`，保存到 `user://achievements.json`。
- `AchievementNotifier`
  监听本地成就解锁信号并显示右上角小型弹窗。
- `WindowDisplayManager`
  应用窗口/全屏设置，并在窗口模式下维持 `16:9` 比例。

交接约定：

- 有生命周期、运行时状态、信号的服务优先使用 Autoload。
- 纯数据、纯计算、一次性转换逻辑继续使用 `RefCounted` static helper。
- 新服务不要直接写进 `player.gd` 或 `main.gd`；先判断是否应该是 `scripts/game/*` flow、Autoload 服务，或 UI 子组件。

## 当前架构护栏

- 战斗成就事件统一通过 `scripts/game/game_achievement_bridge.gd` 转发，`main.gd` 不直接调用 `AchievementService`。
- 敌人击败后的分支处理统一通过 `scripts/game/enemy_defeat_flow.gd`，`main.gd` 不直接处理小 Boss/无尽 Boss 奖励 UI。
- 战斗场景的 UI 创建与玩家信号连接统一通过 `scripts/game/game_scene_wiring.gd`，`main.gd` 不展开具体连接细节。
- HUD 刷新、玩家状态投影、Boss 血条、性能指标统一通过 `scripts/game/game_hud_flow.gd`。
- 角色面板显示/隐藏和暂停规则统一通过 `scripts/game/game_character_panel_flow.gd`。
- 地图边界显示、小地图数据投影统一通过 `scripts/game/game_map_flow.gd`；玩家边界限制统一通过 `scripts/player/player_map_bounds_flow.gd`。
- 故事/无尽模式上下文、故事配队应用、关卡派生参数统一通过 `scripts/game/game_story_context_flow.gd`。
- 平台成就同步只放在 `scripts/achievements/steam_achievement_adapter.gd`，不能混进战斗 flow。
- `scripts/tests/check_architecture_contract.py` 会检查上述边界，防止后续重构时重新耦合。
- `.graphifyignore` 排除了 `addons/godot_mcp/`，避免编辑器 MCP 插件工具函数污染架构图。

## 当前架构特点

- 项目仍然以 `player.gd` 和 `main.gd` 为核心，但已经开始通过 `scripts/player/*`、`scripts/game/*`、`scripts/build/*`、`scripts/ui/*`、`scripts/achievements/*` 拆出边界
- 优点：单人开发推进快，很多内容可以直接实装
- 风险：后续如果继续向 `player.gd`、`main.gd` 直接堆功能，角色逻辑、Build 逻辑、特效逻辑会越来越难维护

## 后续建议

- 不做一次性推倒重写
- 每次新增系统时先补一个小的边界或桥接层
- 优先逐步拆分：
  - `player.gd` -> 角色战斗 / Build / 特效 / 队伍轮转
  - `enemy.gd` -> 普通敌人 / 精英敌人 / Boss 行为
  - `main.gd` -> 战斗装配层 / 具体 flow / 服务桥接；例如敌人击败逻辑已迁入 `scripts/game/enemy_defeat_flow.gd`，UI/信号装配已迁入 `scripts/game/game_scene_wiring.gd`，HUD 投影已迁入 `scripts/game/game_hud_flow.gd`，角色面板流程已迁入 `scripts/game/game_character_panel_flow.gd`，地图展示已迁入 `scripts/game/game_map_flow.gd`，故事上下文派生已迁入 `scripts/game/game_story_context_flow.gd`
