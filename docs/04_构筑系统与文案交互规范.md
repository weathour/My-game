# 构筑系统与文案交互规范

## 当前构筑总框架

- 目前实装的升级 Build 池以“主题”为核心，不再是互不相关的散卡池。
- 初始只开放主题一：`三相荡阵`。
- 主题一固定 3 张核心卡：
  - `起潮`：获得额外攻击手段。
  - `叠浪`：在起潮基础上追加补发段数。
  - `回潮`：改变起潮/叠浪的方向、距离或覆盖面。
- 当主题一内部达到指定配比后，会解锁新主题；每个新主题向卡池追加 3 张牌。
- 每张卡都对剑士、枪手、术师三名英雄都有对应效果，但同一张卡在不同英雄上可以显示不同名字和描述。
- 保留三栏结构：
  战斗栏 / 连携栏 / 技能栏

## 已实装主题与解锁配比

当前主题数据在 `scripts/build/build_database.gd`。

| 主题 | 解锁条件 | 新增卡 | 方向 |
| --- | --- | --- | --- |
| 三相荡阵 | 默认开放 | 起潮 / 叠浪 / 回潮 | 初始单一主题 |
| 万向锋路 | 起潮 Lv.2 + 叠浪 Lv.1 | 贯流 / 分浪 / 环潮 | 单方向推进 -> 扇形 -> 环形/全方向覆盖 |
| 血盾回路 | 起潮 Lv.2 + 回潮 Lv.1 | 饮潮 / 凝盾 / 退潮 | 吸血、护盾、受击反击、击退/控制 |
| 三相终式 | 叠浪 Lv.2 + 回潮 Lv.1 | 蓄相 / 破相 / 归一 | 回能、大招强化、三元素终式 |

解锁由 `scripts/build/build_system.gd` 计算：

- `get_unlocked_theme_ids(card_levels, reward_levels)` 决定当前已开放主题。
- `get_upgrade_pool(slot_id, card_levels, reward_levels, active_role_id)` 根据已开放主题生成可选卡池。
- `record_card_pick()` 会在选卡后检测新主题，并把自动解锁结果写入 `special_reward_levels`。


## 三栏职责

### 战斗栏

- 负责普攻和火力形态变化
- 目标是改变“这名角色怎么打、画面长什么样”

### 连携栏

- 负责切换角色入场、退场、后台协同、轮转收益
- 目标是让切人产生明确节奏收益

### 技能栏

- 负责大招与技能类强化
- 目标是强化大招价值、节奏控制和战术爆发

## 当前构筑问题与共识

- 早期做法是三名角色同时吃到同一套 Build
- 优点：三人都能一起变强，不会只强化到一个人
- 缺点：角色之间容易同质化，看起来只是描述不同，本质效果一致

当前共识：

- 三个人都必须值得切
- 三个人都不残废
- 三个人都能有高光
- 但三个人不是同一种高光

## Build 菜单交互规范

### 当前实装界面

当前升级界面由 `scripts/level_up_ui.gd` + `scripts/ui/components/survivors_card_list.gd` + `scripts/ui/components/survivors_hover_detail.gd` 组成。

- 升级时需要同时选择：
  1. 一张“英雄特性训练”选项。
  2. 一张 Build 卡。
- 卡片本体只显示短标题和简短摘要，避免列表过高。
- 鼠标悬停卡片时显示详情浮层。
- 详情浮层会按内容估算尺寸，受窗口边界限制，移开后消失。
- 列表右侧滚动条强制常显，避免卡片较多时找不到拖动入口。

### 卡牌详情要求

- 技能卡详情必须显示三名英雄各自对应的效果和数值。
- `role_effects` 是详情浮层和技能栏说明的共享数据源。
- 若 `description/detail_description` 已经包含“三英雄对应效果 / 数值”，悬停浮层不能重复追加第二份。
- 同一张卡对不同英雄可以有不同显示名，例如剑士“叠浪”、枪手“散射”、术师“雷相”。

### 卡牌类型

当前卡牌类型由 `CARD_TYPE_BY_ID` 定义：

1. `new_passive`：新的被动技能。
2. `passive_attack`：被动 / 普攻加强。
3. `ultimate`：大招加强。

只有“独立计算冷却时间”的新被动才显示在战斗技能栏。

- 当前进入技能栏的独立冷却被动：`起潮`、`退潮`。
- 无独立冷却的被动、普攻强化、大招强化不单独占技能栏；它们会折叠到普攻或大招的悬停说明中。

## 最终效果卡规范

- 二级菜单右上角只显示主名，例如“天灾化身”
- 进入详情后显示完整名：
  `天灾化身：断神诸恶`
- 详情里只展示需求卡与等级进度
- 已达成的需求高亮，未达成的灰显
- 不写“已激活/未激活”

## 文案硬规则

- 禁止设计师黑话
- 禁止只写空泛气氛词
- 禁止引入新的抽象机制名词去增加理解成本
- 直接写“玩家做什么，会发生什么”

## 文案统一写法

- “开大”统一写成“释放大招”
- 百分比伤害统一写成：
  `伤害为原来的 XX%`
- 二级菜单写“简述效果”
- 三级菜单写“完整效果 + 进化要求”

## 当前构筑方向共识

- 构筑不该只提供数值
- 应该优先提供：
  战斗方式变化
  视觉表现变化
  轮转收益变化
  最终进化效果

## 当前卡池设计注意事项

- 不允许为了凑数而生成重复卡
- 不允许名字很华丽但实际效果重复
- 最终效果卡必须在视觉上有明显辨识度
- 如果卡的收益玩家看不见，优先补演出

## 英雄特性训练

当前升级训练选项不是三维属性，而是三名英雄的特性：

- 剑士特性：提高最大生命和生命自动恢复；每点为剑士普攻伤害提供 +1。
- 枪手特性：提高移动速度和闪避率；每点为枪手普攻伤害提供 +1。
- 术师特性：提高大招能量自动恢复和吸取范围；每点为术师普攻伤害提供 +1。
- 共同致富：三名英雄特性各 +0.35，同时切换英雄冷却时间乘以 90%。该冷却缩减是乘算叠加：第一次 ×0.9，第二次 ×0.81，第三次 ×0.729。

实现入口：

- 规则数值：`scripts/player/roles/role_attribute_rules.gd`
- 存档、迁移、共同致富计数：`scripts/player/player_attribute_flow.gd`
- 升级选项生成和应用：`scripts/player/player_level_flow.gd`
- 切人冷却倍率读取：`scripts/player/player_switch_flow.gd`

兼容说明：旧存档中的 `strength/agility/intelligence`、更早的 `vitality/agility/power` 会迁移到当前英雄特性键。

## 当前交接注意事项

- 新增英雄后，当前 Build 卡池还默认按剑士 / 枪手 / 术师三套 `role_effects` 展示；如果未来允许从更多英雄中任选 3 名，需要把 `CARD_ROLE_NUMBERS` 和 `ROLE_CARD_VARIANTS` 从固定三英雄表改成按队伍角色生成。
- 新增卡时必须同时补：卡类型、主题归属、三英雄显示名、三英雄数值说明、实际效果入口。
- 如果卡没有独立冷却，不要把它塞进技能栏；应补到普攻或大招 hover 说明里。
