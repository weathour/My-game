# 性能优化与验证记录

本文记录当前密集战斗性能优化的设计边界、已落地改动、已知掉帧原因和 PR 验证方式。目标是在不削弱正式玩法密度、不重写核心战斗、不引入 GDExtension 的前提下，降低主线程帧时间尖峰并保留可复现证据。

## 优化边界

本轮优化遵循以下约束：

- 不通过降低无尽模式敌人数量来掩盖低 FPS。
- 不降低技能、弹幕、掉落物和 Boss 的正式玩法设计目标。
- 不做不可逆的大重写；优先拆分热点路径、批处理、缓存和对象复用。
- 不优先引入 GDExtension；当前仍以 GDScript 和 Godot 场景结构优化为主。
- 所有性能结论优先依赖同场景帧时间、慢帧日志、运行态实体规模和玩法等价 smoke，而不是单次主观体验。

## 已落地优化

### 运行态注册表与批处理

- 敌人、敌方弹幕、玩家弹幕和掉落物运行态注册改为以实例 ID 字典维护，保留既有 Array 返回 API，减少高密度场景中的重复线性扫描。
- 简单普通敌人支持场景级批量更新，关闭大量普通小怪各自的物理回调调度。
- 敌方弹幕和掉落物支持场景级批量 tick，减少每个节点单独 `_physics_process` 的调度成本。
- 玩家弹幕加入池化复用，回收后会恢复场景自身导出的默认视觉/碰撞参数，避免复用串状态。

### 刷怪尖峰削峰

- 大批量预警刷怪结束时不再把所有新怪集中实例化到同一帧，而是进入待刷怪队列并按帧预算消化。
- 开发者模式可按普通小怪 archetype 批量生成，用于复现密集战斗；开发者批量生成不受普通敌人运行时上限限制。
- 无尽模式不再根据低 FPS 动态降低普通敌人上限，敌人密度继续由刷怪规则、难度 profile 和波次设计决定。

### HUD 与 UI 热点

- 战斗 HUD 的时间、技能冷却、大招能量、小地图等表现刷新目标为 30 FPS。
- per-frame HUD 改用轻量属性 payload，避免每帧构建角色详情、技能进化需求和完整说明文本。
- HUD Label、冷却槽和能量显示会跳过未变化内容，降低 UI 重绘和字符串拼接成本。
- 升级触发后，祝福发牌延后到下一帧，并先暂停战斗；升级/技能奖励确认后，完整 HUD 刷新、战斗存档和连续待处理升级弹窗恢复会延后合并处理，避免点击同一帧堆叠重活。

### 存档与日志

- 战斗自动存档固定约 2 秒一次，覆盖当前 `run_save.json` 和 `run_save_backup.json`，不会按时间追加历史快照。
- 升级/奖励确认后的补充保存不再立即阻塞点击帧，而是由奖励维护任务延后执行。
- 性能日志开关默认关闭；开启后写入 `user://performance_trace_latest.jsonl`，记录周期样本和慢帧样本。
- 开发者性能面板展示帧时间分位数、SavePeak、实体计数、待刷怪队列、批处理状态和 feature flags。

### 视觉与玩法修复

- 修复“三命诡影”复活过程中目标丢失时卡在地图中央、不可选中且不动的问题。
- 修复“祸月星核”Boss 视觉在重复刷新后被清掉而隐身的问题。
- 批量伤害查询按形状/半径网格边界收集候选，避免一个过大的合并包围盒扫描过多无关目标。

## 之前掉帧的主要原因

这次掉帧不是单一 bug，而是多个主线程热点叠加：

1. **SceneTree 调度压力**：大量敌人、弹幕、掉落物分别跑物理回调，主要卡在 Godot 主线程，不能自然吃满多核，所以会出现 CPU 总占用低但 FPS 低。
2. **线性注册表操作**：运行态对象越多，注册、移除、计数和保存前遍历越容易形成 O(n) 尖峰。
3. **同帧实例化新怪**：大批量怪物同一帧 instantiate、初始化、加入树和设置视觉，会造成新怪出现时瞬间掉到 20 FPS 左右。
4. **HUD 重建过重**：战斗 HUD 曾过于频繁构建完整角色属性、技能需求文本和冷却详情；即使怪不多，长时间游玩也会积累 UI/字符串计算压力。
5. **保存尖峰**：自动保存或奖励后立即保存会遍历敌人、弹幕、经验晶体、回血道具等运行态数组；实体较多时会成为慢帧。
6. **升级/点技能路径堆叠**：升级发牌、应用祝福/技能、完整 HUD 刷新、存档和连续升级弹窗可能挤在确认前后一帧内，造成点击瞬间卡顿。

## 验证方式

每次修改性能相关代码后至少运行：

```bash
./scripts/check_project.sh
/home/weathour/.local/bin/godot-4.6.2 --headless --path . --quit
git diff --check
graphify update .
```

手动验证建议：

1. 无尽模式正常游玩，观察升级、技能奖励、小 Boss/Boss 奖励确认前后是否还有瞬间卡顿。
2. 开发者模式使用普通小怪批量生成复现高密度场景，检查新怪入场是否分帧消化。
3. 开启性能日志数秒，复现后关闭，检查慢帧样本中的 `delta_ms`、`save_peak`、`pending_spawns` 和 batch 计数。
4. 对比三名角色中后期技能释放，确认没有因批处理和池化导致伤害、视觉、掉落或碰撞规则改变。

## PR 评审关注点

- 优先看 `scripts/game/*`、`scripts/player/*`、`scripts/ui/hud/*` 和运行态 registry/pool 相关改动。
- 注意本轮优化没有把低 FPS 作为降低正式敌人密度的条件；如果评审发现敌人数量下降，应视为回归。
- 性能日志是临时排查工具，默认关闭，不应改变正式玩法。
- 自动存档仍默认启用，但写入频率固定且覆盖同一运行存档/备份，文件数量不会随时间增长。
